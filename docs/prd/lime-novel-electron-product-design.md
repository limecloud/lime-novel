# Lime Novel 独立产品方案与代理系统设计

> 版本：0.3
> 更新：2026-04-02
> 用途：为 `lime-novel` 重新定义“以 AI 代理为底层的长篇小说创作系统”
> 状态：产品与技术总纲

---

## 一、结论

`Lime Novel` 不应该再被定义成“带几个写作技能的桌面编辑器”，也不应该沿用 Lime 主 App 的通用创作壳。

它更准确的定义应该是：

**一个本地优先、项目优先、连续性优先、代理优先的长篇小说创作系统。**

这里的“代理优先”，不是把产品做成工程控制台，也不是把作者赶进空对话，而是把底层真正做成一套可持续工作的创作操作系统：

- 前台看到的是项目、章节、设定、修订、发布这些稳定工作面
- 中台运行的是项目总控代理、章节代理、设定代理、修订代理、发布代理
- 底层承接的是技能目录、任务执行、记忆提炼、工具调用、搜索与连接器
- 所有结果都必须回写到正文、设定库、问题队列和发布资产

一句话说清楚：

**Lime Novel 的目标不是“帮你写一段”，而是“陪你把一部小说持续写完、修完、发出去”。**

## 二、背景：从素材到作品

Lime 那条价值跃迁链在小说产品里仍然成立，但必须落成真正的系统结构：

![Lime Novel 创作价值链](./assets/lime-creative-background-model.svg)

在 `Lime Novel` 里，它对应的是：

- `素材`：灵感碎片、句子草稿、设定片段、参考资料
- `信息`：章节摘要、场景卡、人物事实、线索记录
- `知识`：角色卡、地点卡、规则卡、时间线、关系网
- `洞察`：连续性问题、节奏问题、人物失真、伏笔断裂
- `判断`：修订优先级、改写策略、章节推进取舍

## 三、为什么这次要从“技能运行时”升级为“代理系统”

`skills` 仍然重要，但它们不应该再承担产品定义本身。

参考你提供的 `claudecode` 代码结构，这里最值得迁移的不是某个按钮，而是下面六条底层原则。

### 3.1 技能不是提示词，而是受控能力单元

从 `claudecode/src/skills/loadSkillsDir.ts` 可以看到，一个技能至少会声明：

- 描述和触发条件
- 可用工具边界
- 执行上下文
- 可选钩子
- 作用路径
- 可挂接的代理类型

这对 `Lime Novel` 的意义是：

- 技能不再只是“写作模板”
- 技能应该是小说动作的最小可复用单元
- 技能要能带脚本、示例、验证逻辑和资源文件
- 技能要能被不同代理调用，而不是直接暴露成一堆前台入口

所以在 `Lime Novel` 里：

- `代理` 负责理解任务、拆分任务、调度能力、维护记忆
- `技能` 负责完成明确动作，例如“抽取人物卡”“检查视角漂移”“生成发布简介”
- `工具` 负责执行确定性操作，例如“读写正文”“更新卡片”“导出文件”“访问搜索或连接器”

### 3.2 子代理要能分叉，但不能污染主写作上下文

从 `claudecode/src/utils/forkedAgent.ts` 可以看到，成熟的代理系统不是“一条对话做所有事”，而是：

- 主代理保留主任务连续性
- 子代理在隔离上下文中执行特定工作
- 共享可缓存的上下文前缀
- 隔离可变状态，避免互相污染
- 只在必要时把结果回写主线程

对小说产品来说，这非常关键。

因为真正的长篇创作会同时存在：

- 正在写的当前章节
- 后台跑的人物一致性检查
- 后台跑的设定提取
- 后台跑的节奏诊断
- 后台跑的导出准备

如果所有工作都挤在一个对话里，作者会被噪音淹没。

所以 `Lime Novel` 应该明确支持：

- 主写作代理持续服务当前章节
- 设定代理在后台提炼或校验卡片
- 修订代理在后台生成问题队列
- 发布代理在后台准备导出资产
- 这些代理通过任务与结果卡回流，而不是直接抢占正文界面

### 3.3 长篇项目必须自带记忆提炼与上下文压缩

从 `claudecode/src/services/SessionMemory/sessionMemory.ts` 与 `claudecode/src/query.ts` 可以看到，两件事是系统级能力，而不是可选优化：

- 周期性会话记忆提炼
- 自动上下文压缩与摘要边界

这对小说系统几乎是硬需求。

因为长篇小说天然会遇到：

- 项目越来越大
- 章节越来越多
- 设定越来越厚
- 会话越来越长
- 上下文越来越贵

所以 `Lime Novel` 必须内建三层记忆：

- `工作记忆`：当前章节、当前场景、当前选区、当前任务
- `章节记忆`：章节目标、已发生事件、人物状态变化、伏笔状态
- `项目记忆`：系列设定、世界规则、角色长期画像、跨章时间线

同时必须内建两类压缩：

- `轻压缩`：把工具结果、搜索结果、长引用压成可继续工作的摘要
- `重压缩`：当项目上下文过大时，把旧轮次对话折叠成结构化项目记忆

### 3.4 能力要按场景动态激活，而不是一次性全塞进来

从 `claudecode/src/skills/loadSkillsDir.ts` 还能看到两个特别有价值的点：

- 条件技能可以按路径激活
- 动态技能可以在运行时被发现和挂载

对 `Lime Novel` 的启发很直接：

- 写正文时，不应该同时加载整套发布能力
- 看设定卡时，不应该把整套章节推进技能都压进上下文
- 做修订时，应该优先加载问题定位、差异比较和证据回溯能力
- 导出时，才需要挂载平台规则、元数据校验和格式打包能力

所以 `Lime Novel` 的能力装配应该始终遵循：

**少量常驻 + 场景激活 + 结果回写。**

### 3.5 工具编排、任务面板、权限边界都要是一等能力

从 `claudecode/src/services/tools/toolOrchestration.ts`、`claudecode/src/tasks/LocalAgentTask/LocalAgentTask.tsx`、`claudecode/src/Tool.ts` 可以看到，一个成熟代理系统不会把“执行”当成黑盒。

它会明确处理：

- 哪些工具能并发
- 哪些任务在后台执行
- 哪些步骤需要审批
- 哪些结果要保留任务记录与转录
- 哪些代理不能弹权限框，必须自动回退

小说产品里同样需要这种边界感。

例如：

- 自动改写正文前，必须先生成可比较提议，而不能直接覆盖原文
- 发布代理执行平台打包前，需要确认发布范围与元数据
- 大批量设定同步需要可撤销与差异记录
- 外部搜索与引用写回要保留来源和证据片段

### 3.6 对 `Lime Novel` 的最终映射

因此，`Lime Novel` 的底层不应再表述为“一组技能运行时”。

更准确的表述应该是：

**一个以代理为编排核心、以技能为动作层、以记忆为持续层、以工具与连接器为执行层的小说创作系统。**

## 四、产品定位

### 4.1 它是什么

`Lime Novel` 是一个：

- 面向长篇小说、系列小说、连载创作的独立桌面应用
- 以项目为最小工作单元，而不是以会话为最小工作单元
- 以章节推进、设定沉淀、修订闭环、导出发布为核心主线
- 以 `Electron + React + TypeScript` 为桌面技术底座
- 以代理系统为主脑，以技能系统为动作层的创作产品

### 4.2 它不是什么

它不是：

- 一个通用 AI 聊天产品
- 一个“帮你续写一句话”的轻写作插件
- 一个把所有小说动作堆进菜单里的工具箱
- Lime 主 App 里的深写作模式
- 一个把工程概念直接暴露给作者的代理控制台

### 4.3 它和 Lime 主 App 的边界

延续你在 Lime 主仓业务文档里定下的边界：

- `Lime 主 App` 负责文章、研究、浅场景、小说衍生分析、拆书、人物关系分析
- `Lime Novel` 负责长篇小说正文创作、连续性驱动写作、深度修订、发布闭环

这个边界不应该再混。

## 五、核心产品原则

### 5.1 项目优先，不是对话优先

用户进入的是一个小说项目，而不是一个空对话。

### 5.2 正文优先，不是代理优先

代理再强，中央主舞台仍然必须是作品本身。

### 5.3 连续性优先，不是单次生成优先

每一次续写、改写、总结，都必须服从项目长期一致性。

### 5.4 结果可回写，不是只给建议

正文、卡片、问题、导出资产，都必须能被真实写回和追踪。

### 5.5 低打扰优先，不让后台代理打断写作心流

后台代理应该以任务卡、结果卡、风险提示的形式出现，而不是持续抢占主屏。

### 5.6 前台中文、后台抽象

前台用“项目、章节、设定、修订、发布”这些作者语言。
`skills`、`hooks`、`审批模式`、`连接器` 这些只保留在工程层和管理层。

## 六、底层系统分层

```text
作者交互层
  项目首页 / 写作画布 / 设定画布 / 修订画布 / 发布画布

代理编排层
  项目总控代理 / 章节代理 / 设定代理 / 修订代理 / 发布代理

任务与技能层
  模板技能 / 章节推进技能 / 设定提取技能 / 一致性检查技能 / 发布技能

记忆与证据层
  工作记忆 / 章节记忆 / 项目记忆 / 证据片段 / 问题队列 / 版本记录

工具与连接层
  文件读写 / 差异比较 / 导出工具 / 搜索 / 外部资料连接器 / 平台发布连接器

模型与服务层
  推理模型 / 向量检索 / 本地数据库 / 云端同步与托管服务
```

### 6.1 代理层定义

#### 1. 项目总控代理

职责：

- 维护当前项目目标与总体状态
- 决定当前应激活哪些子代理与技能
- 汇总章节、设定、修订、发布的结果
- 负责把后台结果转成前台可理解的任务或建议

何时出现：

- 新建项目
- 切换章节
- 长时间写作恢复现场
- 跨章节修订
- 发布前总校验

#### 2. 章节代理

职责：

- 服务当前章节的续写、重写、压缩、扩写、过渡补写
- 读取当前章节目标、场景结构、近期相关章节摘要
- 调用必要的设定与修订信息
- 输出可应用的段落提议，而不是直接覆写正文

何时出现：

- 正在写章节
- 选区改写
- 场景重排
- 续写下一段

#### 3. 设定代理

职责：

- 从正文反向提取角色、地点、规则、道具、时间线节点
- 校验当前文本是否违反已知设定
- 维护角色画像、关系图和章节引用
- 控制哪些设定对当前写作轮次可见

何时出现：

- 保存章节
- 手动“沉淀为卡片”
- 设定冲突检测
- 角色或地点检索

#### 4. 修订代理

职责：

- 生成问题队列
- 对问题给出证据和影响范围
- 生成多种修订方案
- 维护修订记录、接受/拒绝状态与撤销能力

何时出现：

- 章节完成后复查
- 全书一致性扫描
- 用户打开修订画布
- 用户手动发起“诊断本章”

#### 5. 发布代理

职责：

- 生成导出预设
- 补齐元数据和简介
- 执行格式校验
- 准备平台章节拆分与发布资产
- 将发布结果和反馈回写项目

何时出现：

- 导出前
- 连载准备
- 版本快照
- 书稿打包

#### 6. 研究代理

它不是第一阶段必须，但长期值得预留。

职责：

- 查询题材背景
- 拉取参考资料
- 做事实核查
- 生成可引用证据包

它服务的是“小说写作中的研究动作”，不是产品主舞台。

### 6.2 技能层定义

在 `Lime Novel` 里，技能是代理调用的动作包，不是首页导航。

建议第一批技能分成五类：

- `项目技能`：新建项目、模板装配、恢复现场、系列挂接
- `章节技能`：续写、改写、过渡补写、摘要、场景拆分、节奏推进
- `设定技能`：抽卡、同步、时间线、角色画像、引用分析
- `修订技能`：一致性检查、视角检查、语言重复、节奏诊断、问题生成
- `发布技能`：元数据补全、导出排版、平台拆章、反馈回流

技能目录建议保留“文件夹能力包”设计，而不是单文件提示词。

每个技能包至少包含：

- `SKILL.md`
- 示例或模板
- 必要脚本
- 可选资源文件
- 常见坑位说明
- 输出格式说明

### 6.3 记忆层定义

#### 工作记忆

面向当前几分钟的写作轮次，只保留：

- 当前章节目标
- 当前场景
- 当前选区
- 当前代理正在执行的任务

#### 章节记忆

面向单章或相邻章节，保留：

- 本章摘要
- 人物当前状态
- 本章伏笔与回收状态
- 章节冲突和待修问题

#### 项目记忆

面向整部书，保留：

- 系列与卷册结构
- 世界规则
- 人物长期画像
- 跨章时间线
- 关键关系与主线冲突

#### 证据层

不是“记忆”，而是“为什么得出这个判断”的依据，建议独立存：

- 命中章节
- 原文片段
- 相关设定卡
- 问题来源
- 搜索引用来源

### 6.4 工具、搜索、连接器的职责边界

这里要保持和你前面对 Lime 的判断一致。

#### 工具

负责确定性执行，例如：

- 读写正文
- 更新卡片
- 生成差异
- 导出文件
- 操作本地数据库

#### 搜索

负责不在项目本地知识里的补充信息，例如：

- 题材研究
- 历史、地理、职业常识
- 平台规则
- 外部参考材料

#### 连接器

这里不要把 `MCP` 当成产品叙事。
更准确的说法是“连接器层”。

它负责：

- 接入本地项目知识源
- 接入外部资料库
- 接入托管服务
- 接入发布平台或团队协作系统

在产品层，用户看到的应始终是“接入来源”或“资料源”，而不是某种协议名。

### 6.5 审批与安全

代理系统必须保留明确审批边界。

以下动作建议强制确认：

- 大段覆写正文
- 批量更新设定卡
- 清空或重建时间线
- 覆盖导出产物
- 发布到外部平台
- 删除版本快照

## 七、核心工作流

### 7.1 新建项目

1. 作者选择题材模板或空白项目
2. 项目总控代理创建基础结构
3. 项目技能装配默认卡片类型、章节结构和提示规则
4. 首页画布展示可继续入口，右侧 AI 栏给出下一步建议

### 7.2 章节推进

1. 作者进入章节画布
2. 章节代理读取当前章节目标、近期摘要和关键设定
3. 作者选区续写或发起改写
4. 章节代理生成提议块
5. 作者接受、拒绝或再来一版
6. 保存后，设定代理与修订代理在后台更新卡片和问题队列

### 7.3 正文沉淀设定

1. 作者手动标记，或系统在保存后自动检测
2. 设定代理提取角色、地点、规则、道具、时间线节点
3. 作者在右栏确认是否写入设定库
4. 写入后自动更新引用章节与关系

### 7.4 修订闭环

1. 修订代理扫描当前章、当前卷或整部书
2. 生成问题队列并标注证据与影响范围
3. 作者打开某一问题
4. 中央画布展示原文、差异、关联章节
5. 右栏给出多个修订方案
6. 作者应用、稍后或忽略
7. 修订记录写回项目

### 7.5 发布闭环

1. 发布代理读取导出范围与目标平台
2. 检查元数据、章节顺序、格式风险
3. 生成导出预设和平台化资产
4. 作者确认后执行导出
5. 导出结果、版本号、平台反馈回写项目

## 八、数据模型建议

建议核心对象至少包含：

- 系列
- 小说项目
- 卷册
- 章节
- 场景
- 角色卡
- 地点卡
- 规则卡
- 时间线事件
- 修订问题
- 修订记录
- 代理运行记录
- 任务运行记录
- 记忆快照
- 证据项
- 导出资产

建议存储策略：

- 正文按章节与场景分块
- 卡片与时间线用结构化存储
- 代理运行记录与任务结果单独存档
- 证据片段独立索引，便于回溯和显示来源
- 项目记忆与章节记忆分层存储，避免一锅端

## 九、桌面技术栈建议

### 9.1 推荐技术栈

- `Electron`
- `React`
- `TypeScript`
- 文本编辑器：优先可承载长文档、批注、差异与行内提议块的方案
- 状态层：按“项目状态 / 代理任务状态 / UI 状态”分层
- 本地数据库：优先 `SQLite`
- 全文与向量检索：本地优先，可选云端增强
- 文件导出：统一走主进程服务
- 代理执行：建议独立 `Node` 工作线程或后台进程，避免把所有执行压在渲染层

### 9.2 进程分层

```text
主进程
  窗口、文件、导出、系统能力、连接器调度

预加载层
  安全桥接、受控 API

渲染层
  首页画布 / 写作画布 / 设定画布 / 修订画布 / 发布画布

代理执行层
  项目总控代理
  章节代理
  设定代理
  修订代理
  发布代理

数据层
  项目库 / 记忆库 / 任务库 / 索引库 / 导出资产库
```

### 9.3 不建议直接复用 Lime 当前桌面壳的部分

不建议直接继承下列前台心智：

- 通用聊天首页
- 技能管理页式导航
- 多系统页切换式主入口
- 以工程概念组织用户路径

可以复用的反而是更底层的东西：

- 技能目录与执行边界设计
- 连接器和工具接入思想
- 任务、转录、结果记录方式
- 审批与安全边界

## 十、UI 方向摘要

`Lime Novel` 的 UI 应明确采用：

**左侧小说工作面 + 右侧 AI 双态协作栏**

原因不是为了贴近 Lime 主 App，而是这个结构正好满足小说产品的三件事：

- 左侧必须让作品本身始终占据主舞台
- 右侧必须同时容纳“快速看建议”和“继续多轮对话”两种高频动作
- 后台代理结果必须同时进入建议视图与对话视图，而不是分裂成两套状态
- 默认先打开 `建议`，而 `对话` 视图必须保留常驻输入框，承接深入协作

完整设计见：

- [UI 设计文档](./lime-novel-ui-design.md)

## 十一、MVP 范围

### 11.1 第一阶段必须做

## 十二、阶段路线图

### 阶段 0：架构定盘

## 十三、开源与门槛

如果 `Lime Novel` 选择开源，真正该建立门槛的不是壳层，而是高价值中层。

建议分层处理：

### 13.1 可以开源的层

- 基础桌面壳
- 基础工作面
- 本地项目存储
- 基础代理编排接口
- 基础技能目录协议

### 13.2 适合形成门槛的层

- 高质量小说代理包
- 连续性检测与修订策略库
- 高价值题材研究搜索
- 发布平台适配与反馈回流
- 云端项目记忆与系列知识图谱
- 托管连接器与专业资料源

也就是说：

**开源的是框架，收费的是高价值代理能力、资料网络和长期项目大脑。**

## 十四、最终判断

`Lime Novel` 下一版不该再继续讲成“小说工作台 + 技能运行时”。

更准确的产品定义应该是：

**一个以项目为对象、以章节为主线、以代理为主脑、以技能为动作层、以记忆为持续层的小说创作系统。**

对作者来说，他看到的只是：

- 作品
- 章节
- 设定
- 修订
- 发布

但在这些稳定工作面下面，真正让产品变强的，是一套像 `claudecode` 一样有分工、有记忆、有任务、有边界的代理系统。
